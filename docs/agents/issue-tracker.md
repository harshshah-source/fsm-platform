# Issue tracker: Local Markdown

Issues and PRDs for this repo live as markdown files in `.scratch/`. This repo is not (yet) a git
repository and has no GitHub/GitLab remote, so there is no `gh`/`glab` workflow — operate on files.

## Conventions

- One feature per directory: `.scratch/<feature-slug>/`
- The PRD (if any) is `.scratch/<feature-slug>/PRD.md`
- Implementation issues are `.scratch/<feature-slug>/issues/<NN>-<slug>.md`, numbered from `01`
- Each feature directory may have an `INDEX.md` listing issues in dependency/build order
- Triage state is recorded as a `Status:` line near the top of each issue file (see
  `triage-labels.md` for the role strings); a `Type:` line records HITL vs AFK
- Comments and conversation history append to the bottom of the file under a `## Comments` heading

## Issue file structure

Every issue file uses this skeleton:

```
# NN — <title>
Status: <triage-label>
Type: <AFK|HITL>

## What to build
## Acceptance criteria
## UI surfaces            ← required if the issue touches any user-facing surface; else "n/a"
## Reference              ← required for any issue with UI surfaces; lists the exact image path(s)
## Blocked by
```

**`## UI surfaces`** names each Admin and Mobile surface the issue creates or modifies, e.g.
`Admin: SE Activity board (new)`, `Mobile: n/a`. If a surface is deferred, it must name the
follow-up issue that owns it. Backend/test-only issues record `n/a`.

**`## Reference`** lists the authoritative image(s) by on-disk name, e.g.
`docs/ui/desktop/v2-reference/18-component-requests.png` (mobile files carry a double `.png.png`
extension). `n/a` only if the issue is backend/test-only with no user-facing surface.

Acceptance criteria for any UI surface must be phrased against its reference image (layout,
hierarchy, role visibility) and are subject to the Parity gate in `docs/agents/workflow.md`.

## Active backlog

`.scratch/fsm-platform-v1/` holds the v1 platform backlog of tracer-bullet vertical slices plus
`INDEX.md`, generated by `/to-issues` from the `docs/` truth sources. Issues are sequenced loosely
along the backend LLD phases P0–P7; each issue's "Blocked by" section gives the precise build order.

**`INDEX.md` is the single live source of backlog truth.** Read it before starting work. **Never
hardcode or assume a backlog count** — the set grows as follow-up issues are filed (see below). The
HITL slices are `01` (foundation/infra) and `03` (notifications — external provider provisioning);
the rest are AFK (`ready-for-agent`). See `docs/agents/workflow.md` for what HITL means here.

## Follow-up issues

Follow-up issues are real backlog items, numbered with the next free `NN` (the first follow-ups are
`45`–`48`). They are grouped under a "Follow-ups" section in `INDEX.md`, each carrying a parent
pointer and a short reason (e.g. `45 — Plants Admin UI → 02  (closes AC#2 partial)`). A follow-up
links back to the issue that spawned it and is sequenced by its own "Blocked by".

## Accepted-with-follow-up

When a review finds partial/deferred acceptance criteria, a future dependency, or architecture debt
**but the issue's core objective is met**, mark the parent **accepted** and **do not reopen it**.
Instead:

- File follow-up issue(s) at the next free `NN` and add them to `INDEX.md`'s Follow-ups section.
- Record the disposition (accepted-with-follow-up, what was deferred, follow-up numbers) in the
  issue's progress doc under `docs/progress/`.

Reference pattern: **Issue 02** → accepted with follow-ups `45`–`48`
(`docs/progress/02-org-reference-config-settings.md`).

## When a skill says "publish to the issue tracker"

Create a new file under `.scratch/<feature-slug>/issues/` (creating the directory if needed), using
the next free `NN`. Add a one-line entry to that feature's `INDEX.md` if one exists.

## When a skill says "fetch the relevant ticket"

Read the file at the referenced path. The user will normally pass the path or the issue number
directly (e.g. "issue 16" → `.scratch/fsm-platform-v1/issues/16-*.md`).
