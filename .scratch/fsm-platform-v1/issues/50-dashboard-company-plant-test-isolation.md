## Problem

`apps/backend/test/dashboard-company-plant.e2e-spec.ts` (Issue 06 slice 2) is not isolation-safe and
intermittently fails the full suite:

- It uses **hardcoded** `device_id`s (`9_062_001n..9_062_003n`) instead of namespacing with a per-run
  value (the pattern every newer spec uses, e.g. `const NS = Date.now()`). If a prior run crashes
  before `afterAll`, those exact rows survive and the next run's `prisma.device.create` fails with a
  unique-constraint error.
- Its `afterAll` deletes `device_states` then `devices`, but **not** any `failure_cycles`/`tickets`
  that reference those devices first. If the shared-DB ticket-creation path ever materialised a
  failure cycle for the seeded inactive `device_states`, the device delete fails on
  `failure_cycles_device_id_fkey`, leaving orphans that poison subsequent runs.

## Reproduction (confirmed)

During the Issue 14a full-suite run the file failed with `Unique constraint failed on (device_id)` on
`device.create` and `Foreign key constraint violated: failure_cycles_device_id_fkey` on the
`device.deleteMany` in `afterAll`. Clearing the orphaned rows for `9_062_001..003` (FK-safe order) made
the file pass again — confirming data pollution, not a logic bug.

## What to change

- Namespace the fixture `device_id`s (and the fixed plant names `P-zm2` / `P-other2`) with a per-run
  value so concurrent/leftover rows can't collide.
- Harden `afterAll` to delete in FK-safe order: `ticket_events` → `tickets` → `failure_cycles` →
  `device_states` → `devices` (mirror the teardown in the Issue 11/13 scheduling specs), and wrap so a
  partial failure still cleans the rest.
- Optionally add a shared test helper for FK-safe device teardown to stop this recurring.

## Acceptance criteria

- [ ] Fixture device_ids / plant names are per-run unique (no hardcoded collidable values)
- [ ] `afterAll` tears down dependents before devices, in FK-safe order
- [ ] The file passes in isolation **and** twice in a row without manual DB cleanup
- [ ] No production code change (test-only)
